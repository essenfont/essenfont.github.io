# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) for working in this repository.

## What this repo is

Source for [essenfont.github.io](https://essenfont.github.io) — the public
website for **Essenfont**, the universal Unicode 17 font. The font itself
lives in a sibling repo (`essenfont/essenfont`); this repo is the marketing,
documentation, and Unicode-browser surface for it.

**Stack:** Vite + vite-ssg + Vue Router. Pre-renders ~160,500 routes at
build time (one HTML file per assigned Unicode 17 codepoint, plus per-block,
per-plane, and per-property-value pages).

Sibling repos in the Fontist ecosystem:
- [`essenfont/essenfont`](https://github.com/essenfont/essenfont) — the font
  source, donor registry, and Ruby build pipeline.
- [`fontist/fontist`](https://github.com/fontist/fontist) — package manager
  that installs Essenfont (`fontist install "Essenfont"`).
- [`fontist/fontisan`](https://github.com/fontist/fontisan) — font processor
  used by the build.
- [`fontist/fontist.org`](https://github.com/fontist/fontist.org) — sister
  site whose Unicode browser patterns inspired this one.

## Commands

```bash
npm install
npm run dev          # vite dev server
npm run build        # gen-ssg-routes.mjs → vite-ssg build → dist/
npm run preview      # preview production build
npm run gen-routes   # regenerate public/ssg-routes.json + sitemap.xml only
npm run gen-unicode  # regenerate public/unicode* from unicode.org UCD XML
```

There is no lint script. There is no test script yet.

## Architecture

### Entry chain
`index.html` → `/src/main.ts` (vite-ssg `ViteSSG` factory) → `App.vue` →
`src/layouts/DefaultLayout.vue` (nav + footer shell) → `<RouterView />`
wrapped in `<Suspense>` (pages use top-level `await`).

`vite.config.ts` reads `public/ssg-routes.json` (generated by
`scripts/gen-ssg-routes.mjs`) to populate `ssgOptions.includedRoutes`,
and sets `dirStyle: 'nested'` so routes emit as `foo/index.html` (both
`/foo` and `/foo/` resolve on GitHub Pages).

### Routing
All routes live in `src/router.ts`. Block/char slugs use the `:hex` and
`:blockSlug` params. Catch-all `/:path(.*)*` → `NotFound.vue`. The
`PropertyListPage` / `PropertyDetailPage` components are reused for
four Unicode properties (scripts, category, bidi, combining) via `props`
on the route rather than per-property components.

### SSR-aware data loading
Pages use top-level `await` with helpers from `src/lib/ssr-fetch.ts`:
- `fetchJson<T>(path)` — reads from `public/<path>` via `node:fs` during
  SSG, regular `fetch` in browser.

**Important:** `node:fs` and `node:path` must be dynamically imported
*inside* the `if (isSSR)` branch. Static top-level imports cause Rollup
to reject the build ("resolve is not exported by __vite-browser-external")
because the browser bundle cannot resolve node builtins. Vite tree-shakes
the conditional dynamic import away when `isSSR=false`.

Each domain has its own loader that wraps `fetchJson`:
- `src/lib/unicode/data/loader.ts` — `loadAllBlocks()`, `loadBlockCharacters()`,
  `loadUnicodeVersion()`, `loadBlock()`.
- Pages should never call `fetch()` directly — go through the domain loaders.

### Library boundaries (`src/lib/unicode/`)
Pure TypeScript, no Vue dependency. Barrel export at
`src/lib/unicode/index.ts`. Layering:
- `types.ts` — interfaces only (`UnicodeBlock`, `UnicodeCharacter`,
  `UnicodePlane`, `PlaneKey`, `GridMode`).
- `constants.ts` — pure functions (`hexCp`, `safeChar`, `displayChar`,
  `planeForCodepoint`, `scriptGroup`, `blockSlug`, `blockDisplayName`,
  `categoryName`, `scriptDisplayName`, `bidiClassName`, `combiningClassName`,
  `isControlChar`, `controlAbbrev`, `controlName`, `cEscape`, `charRoute`).
- `data/loader.ts` — uses `fetchJson` from `../../ssr-fetch`.
- `components/UnicodeBlockGrid.vue` — Vue component that renders the per-
  character grid for a block.

### Page conventions
`src/pages/{Domain}{Action}Page.vue` (e.g., `UnicodeCharPage`,
`DownloadPage`). Pages are the composition root. They use top-level
`await` to pre-fetch data during SSG, then call `useHead()` from
`@unhead/vue` for per-page `<title>`, OG tags, canonical link.

### Composables
There are no composables yet. When adding one, follow fontist.org's
pattern: each owns one concern.

### Build pipeline
1. `scripts/gen-ssg-routes.mjs` — reads committed `public/unicode-blocks.json`
   + per-block JSON, emits `public/ssg-routes.json` (list of paths) and
   `public/sitemap.xml`. Produces ~160,500 routes (one per assigned
   codepoint + per block + per plane + per property value). Takes a few
   seconds; the subsequent vite-ssg build takes ~10 minutes due to the
   route count.
2. `vite-ssg build` — pre-renders every route in `ssg-routes.json` to
   `dist/`.

### Unicode data regeneration (separate from build)
The committed `public/unicode-blocks.json`, `public/unicode-version.json`,
`public/unicode/blocks/*.json` (340 files), and `public/unicode/indexes/*`
are reproducible from upstream UCD XML via:

```bash
npm run gen-unicode                                   # Unicode 17.0.0
npm run gen-unicode -- --unicode-version=16.0.0       # pin a version
npm run gen-unicode -- --force                         # re-download
```

Pipeline: `scripts/gen-unicode-data.mjs` → `scripts/lib/ucd-xml.ts` (pure-
logic parser, vendored from `fontist/fontist.org`). Downloads
`https://www.unicode.org/Public/<version>/ucdxml/ucd.all.flat.zip` to
`vendor/ucd/` (gitignored), extracts via `unzip`, parses XML, and
atomically writes the JSON files.

This step is **not** wired into `npm run build` — Unicode data changes
rarely and the download is ~30 MB. Run `npm run gen-unicode` manually
when upgrading the Unicode version, then commit the regenerated files.

### Static data layout (`public/`)
- **Committed to repo:** `unicode-blocks.json`, `unicode-version.json`,
  `unicode/blocks/*.json` (340 block files for Unicode 17.0.0),
  `unicode/indexes/*` (property indexes: by-script, by-category, by-bidi,
  by-combining + per-value detail files), `robots.txt`, `site.webmanifest`,
  `logo.svg`, `logo-full.svg`, `favicon.svg`.
- **Generated at build time** (gitignored): `ssg-routes.json`, `sitemap.xml`.
- **UCD XML cache** (gitignored under `/vendor/`): `vendor/ucd/ucd-<version>.flat.{zip,xml}`.

### Essenfont font file
The actual `Essenfont-Regular.ttf` / `.otf` / `.woff2` binaries are
**never committed**. They are downloaded from
`github.com/essenfont/essenfont/releases/latest` at user-install time.
The website's CSS uses a `var(--spec-font-glyph)` family that resolves
to `"Essenfont", "Noto Sans", sans-serif` — if the user has installed
essenfont locally, the browser uses it; otherwise it falls back.

### Styling
`src/styles/main.css` defines a paper-and-ink palette inspired by
fontist.org's specimen aesthetic and metfont's typographic grid:
- Brand: `--ef-rose: #bf4e6a` (rose accent), `--ef-dark: #1c1a18` (ink)
- Light tokens: `--spec-paper: #f1ece1`, `--spec-ink: #1c1a18`
- Dark tokens (`html.dark`): `--spec-paper: #161513`, `--spec-ink: #ecdfd0`
- Type: Spectral (display), Inter (body), IBM Plex Mono (code)

The `--vp-c-*` CSS variable names are kept for VitePress component
compatibility (in case any future component expects them).

## Logo

The mark is a geometric **E** (three horizontal bars + vertical spine) with
**17** set in the middle bar in the rose accent. Three variants live in
`public/`:
- `logo.svg` — square icon, used as favicon and in the navbar.
- `logo-full.svg` — horizontal lockup (icon + wordmark + "UNIVERSAL · UNICODE 17" tag).
- `favicon.svg` — derived from `logo.svg`.

When editing the logo, keep the three variants in sync — same E, same 17,
same rose.

## Conventions inherited from the broader Fontist ecosystem

These apply across all Fontist sites (root + subsites) — keep them aligned
when changing this repo:

- **Per-page Open Graph tags** via `useHead()` in each page component.
- **`sitemap.xml` + `robots.txt`** — `public/robots.txt` is committed;
  `public/sitemap.xml` is generated at build time from the route list.
- **Directory-style URLs** via vite-ssg's `dirStyle: 'nested'` option.

## Repo-local rules

- See the user's global `~/.claude/CLAUDE.md` for absolute rules that apply
  across all repos: never delete source files; never push tags or push/merge
  to main; never add AI attribution (`Co-authored-by`, "Generated with",
  etc.) to commits or PRs; never hand-roll serialization; follow OCP/DRY/MECC
  and avoid `send` / `instance_variable_get` / `respond_to?` type checks.
- **No AI attribution.** This overrides any default behaviour. Commits and
  PRs must look like normal human work.
- **All changes go through PRs.** Never commit to main, never push tags,
  never merge to main locally without explicit user authorization.
- **Never commit font binaries.** TTF/OTF/WOFF2 are distributed via GitHub
  Releases only.
